# 我每天花 20 分钟做的事，被 3 行字替代了

---

## 一个早晨，和一个念头

六点半的闹钟响过，安全运营工程师刘工（化名）打开电脑，登录 NVD（美国国家漏洞数据库），设置日期筛选，把过去 24 小时新增的漏洞一个个过一遍。CVSS 评分（安全威胁等级，满分 10 分）7.0 以上。逐条复制标题、描述、CWE 分类（通用弱点枚举编号）、补丁链接，贴进报告模板，发给团队。

这套流程他跑了将近半年。每天早上，20 分钟，不多不少。

直到有一天，他随手在 GitHub 刷到一个脚本。别人写的。原理很简单：调 NVD 的接口，抓数据，过滤，生成报告，自动推企业微信。刘工试了一下——跑通了。

然后脑子里冒出一个念头，不是"这个脚本真方便"，而是：

**它做的事，我每天也在做。那我自己能不能也做一个？**

---

## 抄作业，是正经事

在软件工程领域，有个说法叫"不要重复造轮子"。意思是已经有现成方案了，就别自己重新发明一遍。这个道理说出来大家都懂，但真到了实操层面，"抄别人的方案"这件事，总是让人有点心虚。

刘工也心虚过。后来他想明白了：**借鉴别人验证过的有效方法，是正经的工程思维，不丢人。**

区别在于"抄"的方式。有人原封不动拿来用，用完就忘；有人拿过来，改一改，塞进自己的工具箱，变成自己工作流的一部分。刘工是后者。他没有直接用那个 GitHub 脚本，而是在理解它的原理之后，把里面的代理逻辑、数据流向、报告格式单独拆出来，重新组装成了自己的东西。

这一步其实很关键。他没有把脚本直接拿来用，而是把里面的逻辑拆开，理解了，再重组。这个过程本身就是学习——前者是在理解之后的主动选择，后者是不假思索的拿来主义。前者才能变成你的能力。

> 在国内，这个逻辑其实很常见。早期那批团购网站，抄的是谁不重要，重要的是抄过来之后变成了自己的地推铁军；推荐算法这套东西，早期多少借鉴了搜索巨头的思路，后来做大了也没人在意最初的来路；淘宝小卖家入门第一课，几乎都是"去抄头部店铺的视觉"——只不过大家不愿意承认最初的路子是这么走出来的。

## 那两步之后，他想明白了

回到刘工的故事。他后来花了两天时间，把脚本改成了自己的 Skill。

改完之后，他只需要对着电脑说一句"今日 CVE"，电脑自动拉数据、过滤高危项、生成报告、写入文件。整个过程实测三秒。

但如果只看到这里，你只读到了故事最表面的那一层。

**真正发生的事，是刘工把自己脑子里每天都在执行的那套流程，提取出来，变成了可以被语言触发的东西。**

这个区别很微妙，但它解释了一件更重要的事：

我们每个人，每天都在无意识地做"模式提取"这件事。

比如，你每天出门前会检查手机、钥匙、门卡。你不需要刻意想这件事，身体自动完成。再比如，你每个月发工资之后，会先把还债的钱转走，再把生活费分到不同的账户，这也是大脑自动提取出来的模式。

这些模式存在了，但它们存在于你的"隐性知识"里——你做得到，但你说不清楚是怎么做到的。你没法把它交给别人，也没法在需要的时候快速调用它。

**Skill 的本质，就是把隐性知识变成显性工具。** 你把那个"说不清楚但做得到"的事描述出来，AI 就能替你执行。执行一万次，每次都是零认知负荷。

这是第二个洞见，比第一个更深一层。

## 一个意外发现的反转

刘工的故事讲完了，但我还想多说一件最近让我重新审视这件事的观察。

刘工后来把自己的 Skill 分享给了团队里的其他人。几个人用了三个月之后，有人提了一个反馈："我现在遇到重复的事，会习惯性地先想，这件事能不能写成 Skill。"

这句话看起来很简单，但它指向了一个很实际的转变：**不是 AI 替代了人的工作，而是人开始重新审视自己每天在做什么。**

这让我想起后来留意到的一个现象：那些对 AI 最焦虑的人，往往不是最容易被替代的人，而是从来没有系统化审视过自己工作流程的人。因为他们不知道自己每天在做什么，自然也不知道哪些部分可以交给 AI。

反过来，**那些已经在主动整理自己工作流程的人，往往对 AI 的态度更从容**——因为他们清楚，AI 能接管的是模式，不是判断。

这个现象指向一个值得停下来想一想的事实：

> **焦虑来自模糊，从容来自系统化。**

## 普通人能怎么做

说了这么多，刘工毕竟是安全工程师，工作内容离大多数人的生活有一定距离。

但"模式提取"这件事，不分职业。你每周要整理一次周报，格式固定，数据来源固定，逻辑固定——这是一件每周都在重复的事。你每次做演示前要手动格式化 PPT，字体、配色、标题位置，每次都在重复同样的操作。你每次写完文章要检查一遍错别字，流程固定，但每次都手动做。

这三个场景，说的都是同一件事——你比你以为的更接近自动化。

**你不需要懂代码才能做到这件事。** 你只需要做到一件事：

> 找一件你本周重复做过三次以上的事，用五分钟时间，把它的步骤写在纸上。不要写代码，就写大白话。写完之后问自己：这件事里，有哪些步骤是每次都一样、不需要动脑子就能完成的？

那些"不需要动脑子"的步骤，就是你潜在的第一个 Skill。

## 一个值得今天就问的问题

写到最后，我想把刘工的故事收在一个更小的地方。

刘工跑了半年每天 20 分钟的手工流程，才在 GitHub 上偶然看到一个脚本。如果他早点意识到这件事可以自动化，那半年就不需要那样度过。

但这还不是我想问的问题。

我想问的是：

**你昨天做的哪件事，是前天做过的？上周做的哪件事，是上个月做过的？你有没有停下来想一想，其中哪一件，其实只需要花五分钟写下来，就能让以后的每一天都省掉？**
